### Q1: Language-Agnostic Database, Crawler Structure, AI Descriptions & Cross-Language Linking

*From: Study Guide §1.2 (AST / Translation Unit)*

> The AST can be run on any language code and stored in a translation unit with its elements. Does our database store this info? How does parsing go to individual crawler folders? Where does the AI description sit? Can we create links between nodes of any language?

**Tag**: 🔧 **IMPROVEMENT**

**Answer**:

**Database is fully language-agnostic.** The `Function`, `calls`, `File` tables have zero language-specific columns. A C function and a Python function are both just rows in the same `Function` table. The `File` table has a `language` column — that's the only place language is recorded.

**Crawlers live in separate files** under `codecrawler/crawlers/` (one per language). Each implements `BaseCrawler` ABC and outputs the universal `ParseResult` DTO. The pipeline uses `LANGUAGE_MAP` (extension → language string) to route files to the correct crawler. Adding a new language = add one new crawler file + register it.

**AI descriptions are columns in the Function table:**
- `summary` (TEXT) — LLM-generated natural language description
- `embedding` (FLOAT[384]) — vector for semantic search
These are *enrichment layers* on top of structural parse data. Both human-readable (summary) and machine-readable (embedding) representations live side by side.

**Cross-language links are architecturally possible** because all functions share the same table and all call edges share the same `calls` table. A call from a C function (id=1) to a Python function (id=2) is just `INSERT INTO calls (1, 2, line, true)`. **However, v4 doesn't detect these yet** — each crawler works in isolation.

**Design Impact**:

v4.5 should add **Cross-Language Edge Detection** — the ability to detect and record call edges that cross language boundaries:
- C → Python: detect `PyObject_CallFunction("func_name", ...)` patterns
- Python → C: detect `ctypes.CDLL` / `cffi` patterns
- Shell → C: detect binary invocations that map to indexed C programs
- C → Shell: detect `system("script.sh")` / `popen()` patterns

This requires each crawler to emit **"outbound foreign call hints"** — partial `CallEdge` objects where the callee is in a different language. A new **`CrossLanguageLinker`** post-processing stage would resolve these hints against the full function index.
